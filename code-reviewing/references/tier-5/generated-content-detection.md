# Generated Content Detection

**Tier:** 5 (Generated Content)
**Load When:** Large changesets (50+ files) or when generated patterns detected
**Token Cost:** ~1,200 tokens

## Overview

This reference provides detection patterns for identifying files that were generated by scripts or tools. When detected, the code review should focus on the generator (source of truth) rather than the generated output.

## Why Detect Generated Content?

1. **Review the source of truth**: The generator script is what matters, not its output
2. **Reduce review noise**: Large changesets often include many generated files
3. **Identify stale generation**: Generated-only changes may indicate outdated regeneration
4. **Prevent false positives**: Generated code patterns may look like issues but are intentional

## Detection Markers

### Markdown Files

| Pattern | Location | Example |
| --- | --- | --- |
| `Generated: YYYY-MM-DD` | First 5 lines | `Generated: 2025-12-29 15:42:18` |
| `<!-- Generated by ScriptName -->` | Last 10 lines | `<!-- Generated by Export-CoverageMatrix.ps1 -->` |
| `Auto-generated` | Anywhere in header | `# Auto-generated Report` |

**Regex patterns:**

```regex
/Generated:\s*\d{4}-\d{2}-\d{2}/
/<!--\s*Generated by\s+([^\s>]+)/
/^#.*Auto-generated/m
```

### JSON Files

| Pattern | Location | Example |
| --- | --- | --- |
| Root `"timestamp"` field | Top level | `"timestamp": "2025-12-29T00:52:52.404Z"` |
| `"generated_at"` field | Top level | `"generated_at": "2025-12-29T00:52:52Z"` |
| `"generator"` field | Top level | `"generator": "Export-Targets.ps1"` |

**Detection logic:**

```text
1. Parse JSON (if valid)
2. Check for root-level "timestamp", "generated_at", or "generator" fields
3. ISO-8601 format indicates machine generation
```

### Script Files (PowerShell, Bash, Python)

| Pattern | Location | Example |
| --- | --- | --- |
| `Auto-generated from` | Comment block | `# Auto-generated from task file` |
| `DO NOT EDIT` | Early comments | `# DO NOT EDIT - regenerate with script.ps1` |
| `Generated by` | Header | `# Generated by Update-VerifyScripts.ps1` |

**Regex patterns:**

```regex
/Auto-generated from/i
/DO NOT EDIT/i
/Generated by\s+(\S+)/i
```

### Code Files (C#, TypeScript, etc.)

| Pattern | Location | Example |
| --- | --- | --- |
| `// <auto-generated>` | File header | .NET code generator marker |
| `// Auto-generated` | File header | Generic auto-gen marker |
| `@generated` | File header | Common generator annotation |
| `*.g.cs`, `*.generated.ts` | Filename | Generated file extensions |

**Regex patterns:**

```regex
/\/\/\s*<auto-generated>/
/\/\/\s*Auto-generated/i
/@generated/
```

**Filename patterns:**

```glob
*.g.cs
*.generated.cs
*.g.ts
*.generated.ts
*.auto.json
```

## Detection Algorithm

### Efficient Scanning

To minimize token usage, use efficient scanning:

```text
1. Read first 20 lines of file (head)
2. Read last 10 lines of file (tail)
3. Check against all marker patterns
4. For JSON: parse only if extension is .json
5. For filenames: check before reading content
```

### Full Detection Workflow

```text
For each changed file:
1. CHECK FILENAME first:
   - If matches *.g.*, *.generated.*, *.auto.* → Mark as generated
   - Extract potential generator from naming convention

2. CHECK CONTENT (first 20 + last 10 lines):
   - Apply regex patterns for file type
   - Extract generator name if present in marker

3. IF GENERATED marker found:
   a. Record: file path, marker type, generator name (if known)
   b. Search for generator script
   c. Add generator to review scope
   d. Mark generated file for skip/light review

4. IF NO marker but suspicious:
   a. Check if in typical generated directories
   b. Check for timestamp patterns in content
   c. Flag for manual review if uncertain
```

### Generator Discovery

When the marker identifies a generator name:

```text
1. EXACT MATCH search:
   - Glob: **/{GeneratorName}
   - Example: **/Export-CoverageMatrix.ps1

2. PREFIX search (common patterns):
   - Export-*.ps1 → generates reports
   - Update-*.ps1 → regenerates existing files
   - New-*.ps1 → creates new files from templates
   - Add-*.ps1 → augments files with metadata

3. DIRECTORY search:
   - Look in common script directories
   - Check for scripts with similar base names
```

When no generator name in marker:

```text
1. Check git blame for file creation
2. Look for scripts in same directory or parent
3. Search for scripts that reference the output path
4. List candidates for human review
```

## Edge Cases

### 1. Generator Also Changed

**Scenario**: Both `Export-Report.ps1` and `REPORT.md` are in changeset.

**Action**: Review generator only, skip generated file.

**Output**:

```markdown
- `REPORT.md` - SKIPPED (generator Export-Report.ps1 in changeset)
```

### 2. Only Generated Files Changed

**Scenario**: `REPORT.md` changed but `Export-Report.ps1` did not.

**Action**: Warn user - may indicate manual edit or stale regeneration.

**Output**:

```markdown
**WARNING**: Generated file changed without generator change.
- `REPORT.md` - May need regeneration or was manually edited
- Generator: Export-Report.ps1 (unchanged)
```

### 3. Generator Not Found

**Scenario**: File has generated marker but generator script not found.

**Action**: Review the generated file anyway, note uncertainty.

**Output**:

```markdown
- `output.json` - REVIEWING (generator "unknown-script.ps1" not found)
```

### 4. Manual Edits to Generated File

**Scenario**: File has generation marker but also shows manual edits.

**Detection**: Check git history for non-generation commits.

**Action**: Flag as drift warning.

**Output**:

```markdown
**WARNING**: Generated file has manual modifications.
- `REPORT.md` - Has manual edits since last generation
- Risk: Edits may be lost on next regeneration
```

### 5. Security-Sensitive Generated Files

**Scenario**: Generated file contains credentials, connection strings, or secrets.

**Action**: Still review for security, even if generated.

**Patterns to watch:**

- `*.config.generated.*`
- Files with `password`, `secret`, `key` in content
- Connection strings in generated config

## When to Still Review Generated Files

Always review generated files when they:

1. **Contain secrets or credentials** (connection strings, API keys)
2. **Are security-sensitive** (auth config, permission files)
3. **Have no identifiable generator** (review as normal code)
4. **Show manual modifications** (potential drift)
5. **Are executable** (scripts, binaries that could be malicious)

## Integration Notes

### Workflow Position

Generated content detection should run **after file counting** but **before main review**:

```text
1. Count files
2. Detect generated content ← HERE
3. Identify generators to review
4. Run main code review (excluding generated files)
5. Consolidate findings
```

### Interaction with Other Tiers

- **Tier 1 (Universal)**: Still applies to generators
- **Tier 2 (Language-specific)**: Apply to generators
- **Tier 3 (Security)**: Apply to security-sensitive generated files
- **Tier 4 (Claude Code)**: Apply if generators are CC components
- **Tier 5 (Generated)**: This tier - detection and redirection
